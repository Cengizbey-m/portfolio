# TheTripMan (Client)

**Live site:** https://www.thetripman.com

## Problem & users

The client needed a fast, mobile-friendly website that converts visitors into bookings/leads without exposing operational details. Users are customers looking for transportation services and a simple way to request availability and pricing.

## Security & privacy considerations

**Confidentiality:** production code remains private. This case study focuses on architecture decisions and delivery work without exposing business-sensitive details.

- No client credentials in the repo (use environment variables)
- Least-privilege access for any admin endpoints
- Basic abuse controls (rate limit, bot mitigation where appropriate)
- Avoid exposing internal identifiers or operational data in the UI

## Deployment pipeline (CI/CD + hosting)

- PRs: Vercel preview deploy for quick review
- Main branch: production deployment after checks
- Quality gates (recommended): typecheck, lint, basic smoke test

## Database / data layer

Production can use Postgres (Neon/Supabase) for:

- booking requests (status, timestamps, service type)
- audit trail of contact submissions
- admin content metadata (optional)
